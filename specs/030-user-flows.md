# User Flows

## Основные пользовательские сценарии

### 1. Управление командами

Пользователь открывает `/teams`, где:

- просматривает список команд;
- создаёт новую команду;
- редактирует существующую команду;
- удаляет команду;
- задаёт parent-child структуру через `parent_id`.

### 2. Управление периодами

Пользователь открывает `/periods`, где:

- просматривает список периодов;
- создаёт новый период;
- редактирует период;
- меняет порядок периодов через move up / move down.

### 3. Просмотр OKR команды

Пользователь открывает `/teams/{teamID}/okr` или `/teamOkrs`.

Текущая реализация страницы team OKR работает как:

- SSR отдаёт каркас страницы;
- данные команды и целей подгружаются через API;
- пользователь видит:
  - команду;
  - период;
  - статус периода;
  - суммарный прогресс;
  - список goals;
  - KR с прогрессом и комментариями.

### 4. Работа с целью

Пользователь может:

- создать goal для команды в периоде;
- редактировать поля goal;
- менять вес goal;
- менять порядок goal;
- комментировать goal;
- расшаривать goal на другие команды;
- менять weight goal для owner team или shared team.

### 5. Работа с key result

Пользователь может:

- создать KR для goal;
- редактировать KR;
- менять порядок KR;
- комментировать KR;
- обновлять прогресс KR в зависимости от типа:
  - `PERCENT`
  - `LINEAR`
  - `BOOLEAN`
  - `PROJECT`

### 6. Обновление статуса периода команды

Пользователь может обновить `team period status` для команды и периода.

Поддерживаемые значения:

- `no_goals`
- `forming`
- `in_progress`
- `validated`
- `closed`

## Текущее UX-поведение

На текущий момент UI и backend устроены так:

- часть страниц рендерится через SSR;
- часть данных и действий выполняется через `/api/v1`;
- team OKR page использует SSR-shell + API hydration;
- goal / KR actions доступны через API и формы.

## Важное ограничение текущей реализации

На текущий момент lifecycle-ограничения периода описаны в домене, но не зафиксированы сервером как строгие правила доступа.

Это означает:

- статус периода хранится и обновляется;
- значение статуса валидируется;
- но backend пока не запрещает structural edits goal / KR только на основании `validated` или `closed`.

Поэтому любые UX-ограничения для `validated` / `closed` сейчас должны считаться целевым поведением интерфейса, а не полной серверной гарантией.

## Требование к новым UX-фичам

Любая новая UX-фича должна явно указывать:

- на какой странице она живёт;
- работает ли она через SSR refresh, API hydration или API mutation;
- какие empty / error / loading states обязательны;
- зависит ли поведение от `team period status`;
- проверяется ли ограничение только в UI или ещё и на сервере.
